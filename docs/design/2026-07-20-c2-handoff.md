# (c)2 handoff — where the readback return path stands (2026-07-20)

Orientation for the next session picking up **(c)2's readback return path**. Read this first, then the
linked docs. It says what is *done and shipped*, what the *open problem* is, and — importantly — which
approaches are **already-proven dead ends** so they are not re-attempted.

## The one-line state

An unmodified Vulkan app (`rayland-icosa-cpu`) renders across a real network (C = apollo → S = dop561)
and reads its frames back **10/11 runs perfectly clean**; a **~1/11 residual** delivers a whole
previous frame (`N == N−1`). The residual is the **hard half of the return path**, and it lives in
**virglrenderer's completion-fence semantics** (`T2 < T4`), not in the relay ordering.

## What is done and shipped (do not redo)

- **The defect was diagnosed and a wrong committed conclusion was inverted.** The stale frames are a
  **readback-completion lag on S**, *not* a forward mapped-blob relay race. Proven with an independent
  forward-input (uniform) witness: at every stale frame the resident uniform was already frame N while
  the delivered image was frame N−1. See
  [`2026-07-19-c2-true-remote-mapped-sync.md`](2026-07-19-c2-true-remote-mapped-sync.md).
- **The readback-completion gate landed and is pushed** (`crates/rayland-s/src/delivery.rs` +
  `progress_thread` in `crates/rayland-s/src/main.rs`): a delivery completes only once
  `take_app_blob_writes` shows the readback blob actually advanced. It took the rate from
  *most-runs-losing-1–4-frames* to **10/11 clean**. Design:
  [`2026-07-19-c2-readback-completion-gate.md`](2026-07-19-c2-readback-completion-gate.md).
- **The two-machine oracle is committed:** `scripts/c2-icosa-two-machine.sh` (runs N times, asserts 0
  stale vs native-on-S; PID-based cleanup — **never `pkill`**). It intermittently FAILs on the known
  residual; that is expected, not a regression (its header says so).

## The open problem — and the two dead ends

The residual: with Venus feedback disabled (the only config that renders over a real network), the app's
`vkWaitForFences` is released by the ring **`head`** (`vn_ring_wait_seqno`), which `progress_thread`
ships in step 1 *before* the gated readback in step 2 — so the app can read its own local `res6` on C
before that frame's readback `BlobData` applies.

**Dead end 1 — Direction A (release ordering): built, reviewed, REGRESSED. Do not re-attempt as
designed.** Holding the head-advance until after the readback ships made it *worse* (~3 stale/run).
Conclusive root cause: the step-2 **empty** state is ambiguous between a **copy** submit (must
release the head — the app's upload fence is head-based; proven by no-release → **deadlock + SIGABRT,
119/120 missing**) and a **draw whose readback DMA has not yet landed** (must hold — release → stale).
They are indistinguishable because **the completion fence does not reliably guarantee the readback is
visible when it retires** (`T2 < T4`). Full analysis and evidence in the "Outcome" section of
[`2026-07-20-c2-readback-release-ordering.md`](2026-07-20-c2-readback-release-ordering.md).

**Dead end 2 — Direction B (fence feedback over the network): pre-existing gap, not a gate bug.** With
fence feedback *enabled* over a real link the app **SIGABRTs at frame 0** (the completion never
reaches it). An A/B against the pre-gate baseline proved the gate is innocent — the 2026-07-17
fence-feedback buy-back was only ever validated on **loopback**. Making it work over a real network is
its own large track.

## Where the next attempt should start: the fence (`T2 < T4`)

The real blocker is that `wait_for_work_retired` (`crates/rayland-engine/src/virgl.rs`:
`wait_for_context_fence` → `virgl_renderer_context_create_fence` + `virgl_renderer_context_poll`)
**retires before the readback DMA is host-visible** a small fraction of the time. Background:
[`2026-07-17-return-path-completion.md`](2026-07-17-return-path-completion.md) §8 (`T2 < T4` is real,
0.33–20 ms) and [`2026-07-19-c2-ringidx-decode.md`](2026-07-19-c2-ringidx-decode.md) §8 (the fence is an
empty `vkQueueSubmit` over the render-server's context-op socket, a path that can overtake the ring
thread; the submit-dispatch trigger shrank but did not eliminate this over a real link).

If "the fence has retired" could be made to **reliably imply "the readback is host-visible,"** then
step-2 empty would unambiguously mean *copy*, and a release-ordering fix (Direction A's cap mechanism)
would work. Concrete first investigations for the next session:

1. **Quantify it.** Instrument S's `progress_thread`: on a post-fence *empty* poll, re-read the
   readback blob for a few ms and record whether/when it changes. This directly measures the
   empty-is-actually-a-draw rate and the residual DMA lag (informs whether a bounded settle is viable).
2. **Understand the fence path.** Read virglrenderer's `vkr_context_submit_fence` / render-server fence
   handling: does the context fence FIFO-follow the app's queue submit and cover its readback copy, or
   can it retire earlier? This decides whether a code-side ordering fix exists or a virglrenderer-level
   change is needed.
3. **Note the content-based signal was already tried and is race-prone** (`2026-07-19-c2-ringidx-decode.md`
   §8) — do not re-derive it from scratch; start from why it failed.

## How to run the proof

```
scripts/c2-icosa-two-machine.sh 5        # one batch of 5; run several batches for ≥20 runs total
```
Needs ssh to apollo (C) and this host as S (dop561), a build of `-p rayland-c -p rayland-s -p
rayland-icosa-cpu` (the script builds release). Judge over many runs — the residual is ~1/11. A single
long invocation can be wall-clock-killed; batches of ≤5 are safe. Never `pkill`; kill only by captured PID.
